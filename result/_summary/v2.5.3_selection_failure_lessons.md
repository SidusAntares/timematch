# v2.5.3 checkpoint selection 失败总结与经验教训

## 1. 版本目标

v2.5.3 的目标是解决 v2.5.0 发现的一个关键问题：

> 不同 source-target pair 对 source checkpoint epoch 的偏好不同，固定 source epoch 会损失域适应性能。

因此这一阶段尝试构造一个 target-side checkpoint selection algorithm：

```text
source checkpoint bank
-> target-side unsupervised / weakly grounded selection
-> selected checkpoint 进入 TimeMatch DA
```

最初希望 selection 尽量不依赖 target label，只通过 target 上的无监督信号判断哪个 source checkpoint 更适合当前任务。

v2.5.0 的 30/50/70/100 checkpoint bank oracle 结果提供了上界参考：

```text
v2.5.0 oracle avg = 0.6891
```

这个 oracle 不是一个可直接部署的方法，因为它使用 test label 找最优 checkpoint，但它说明 checkpoint selection 方向确实有潜力。

## 2. 版本链路与结果

| 版本 | 核心做法 | 结果 | 判断 |
|---|---|---:|---|
| v2.5.3 / v2.5.1 类选择器 | coverage / confidence / agreement / entropy 等无监督指标加权 | v2.5.1 avg 约 0.6518，命中 2/12 | 失败 |
| v2.5.3b | temporal perturbation consistency，clean vs time mask / temporal jitter / value noise | avg 0.6638，exact 5/12，near 7/12 | 有信号但不足 |
| v2.5.3c | 对 perturbation trajectory 做 late-gain 连续惩罚 | 修正 32VNH->33UVP，但误伤 33UVP->32VNH | 负结果 |
| v2.5.3d | late-gain 阈值外软惩罚 | 仍选错 33UVP->32VNH，未解决核心问题 | 负结果 |
| v2.5.3e | 纯 perturbation score | exact 2/12，avg 约 0.6467 | 失败 |
| clean v2.5.3f/g | 纯 perturbation + late reject / margin tiebreak | divergence7 上 exact 2/7，gap -0.0673 | 失败 |
| 误实现 robust f/g | 实际使用 robust score + late/margin 逻辑 | exact 6/12，near 8/12，bank avg 0.6779 | 当前 selection-only 最好，但实现命名混淆 |
| v2.5.3h | 0.7 robust + 0.3 pure perturb blend | exact 5/12，near 7/12，avg 0.6762 | 不如 robust f/g |
| v2.5.3i | 完整流程：重新 source train，20-100 每 10 epoch，robust-primary + perturb tie-break，完整 DA | full DA avg 0.6473 | 负结果 |

## 3. 关键失败现象

### 3.1 confidence / coverage 类指标会奖励“自信地错”

早期选择器倾向选择 coverage 高、confidence 高、teacher-student agreement 高的 checkpoint。

但 hard target 上，高 confidence 不一定意味着可迁移性好，可能只是模型更稳定地预测错。

典型任务：

```text
31TCJ -> 32VNH
v2.5.1 / v2.5.3 早期版本倾向选后期 checkpoint
真实 oracle 更偏 ep50
```

结论：

> confidence / coverage 不能作为 checkpoint selection 主信号。

### 3.2 temporal perturbation 有信号，但单独使用不够

temporal perturbation consistency 的想法是：

```text
如果一个 checkpoint 真正学到 target 上稳定的时序结构，
那么 time mask / temporal jitter / value noise 下预测应保持一致。
```

这个信号确实修正了一些任务。例如：

```text
32VNH -> 33UVP
v2.5.3b/c 能把选择从错误后期 checkpoint 拉回 ep30
```

但纯 perturbation 在 12 任务上表现很差：

```text
v2.5.3e exact = 2/12
clean v2.5.3f/g divergence7 exact = 2/7
```

结论：

> perturbation consistency 可以作为辅助信号，但不能单独作为主排序。

### 3.3 trajectory penalty 容易误伤好 checkpoint

v2.5.3c 尝试惩罚 late-gain：

```text
score = base_score * (1 - alpha * late_gain_ratio)
```

它能过滤一些“最后才突然稳定”的异常候选，但也会误伤本来正确的早期 checkpoint。

典型任务：

```text
33UVP -> 32VNH
oracle = ep30
ep30 base perturbation score 很高
但 late-gain penalty 把 ep30 压低，最终选错
```

结论：

> 连续惩罚式 trajectory rule 太粗暴，不能稳定泛化。

### 3.4 robust score 是目前最强静态信号，但仍不够可靠

误实现的 f/g 版本实际上使用了 robust score：

```text
robust score = perturbation consistency
             + class balance
             + shift stability
             - collapse penalty
             - source-prior divergence
```

它得到目前最好的 selection-only 结果：

```text
exact = 6/12
near <= 1pt = 8/12
bank estimated avg = 0.6779
oracle avg = 0.6891
gap = -0.0111
```

这说明 robust score 比 pure perturbation 更有信息量。

但它仍有明显失败，例如：

```text
33UVP -> 32VNH
robust 倾向 ep100
oracle 为 ep30
```

结论：

> robust score 是当前最好的 zero-shot selection 信号，但不足以形成稳定方法。

### 3.5 linear blend 不解决冲突

v2.5.3h 尝试：

```text
score = 0.7 * robust + 0.3 * pure perturb
```

结果不如 robust f/g：

```text
v2.5.3h exact = 5/12
near = 7/12
gap = -0.0128
```

主要问题是 robust 和 perturbation 的数值不完全可比。线性混合无法可靠表达“什么时候该相信 robust，什么时候该相信 perturbation”。

结论：

> 不要继续调 blend 权重。问题不是 0.7/0.3 没调好，而是两个信号不适合线性混合。

### 3.6 v2.5.3i 完整流程暴露了 ep20 和 tie-break 的风险

v2.5.3i 做了一次更接近正式流程的实验：

```text
source train 100 epoch
保存 20,30,40,50,60,70,80,90,100 checkpoint
robust-primary + perturb tie-break selection
selected checkpoint 完整 DA
```

最终结果：

```text
v2.5.3i fullflow avg = 0.6473
```

关键问题：

```text
11/12 个任务触发 perturb_tiebreak
只有 1/12 真正走 robust_primary
```

这说明：

```text
robust top1/top2 gap <= 0.01
```

这个 tie-break 条件过宽，导致 robust 主排序几乎失效。

同时 ep20 被频繁选中，污染候选池：

```text
31TCJ -> 32VNH selected ep20, F1 = 0.5576
32VNH -> 31TCJ selected ep20, F1 = 0.5170
33UVP -> 31TCJ selected ep20, F1 = 0.5829
```

最严重失败：

```text
32VNH -> 33UVP selected ep50, F1 = 0.3987
old oracle ep30 = 0.6674
```

结论：

> 增加候选密度并不一定提升选择质量。过早 checkpoint 和过宽 tie-break 会让 selection 不稳定。

## 4. 关于完整 DA 波动的判断

我们对比过同一 checkpoint 的重复 DA 结果：

```text
16 个对齐样本中，15 个差异小于 1 个点
只有 31TCJ->32VNH 在 v2.5.3b 出现一次 -7 点异常
```

因此 selection 开发阶段可以主要看：

```text
最终选择了哪个 checkpoint
```

而不必每次都跑完整 DA。

但当 selector 接近可用时，仍需要一次 full DA 验证最终性能。

## 5. 主要经验教训

### 5.1 zero-shot target-side selection 信号弱

v2.5.3 的核心教训是：

> 不经过 target 上真实适应过程，仅靠静态无监督指标很难稳定判断哪个 checkpoint 对当前 target 最优。

这些指标都存在问题：

| 指标 | 问题 |
|---|---|
| confidence / coverage | 会奖励自信错误 |
| teacher-student agreement | 容易反映模型自洽，而不是迁移质量 |
| class entropy / balance | 只能检测塌缩，不能判断类别语义正确 |
| source prior JS | source-target 类分布本来就可能不同 |
| shift stability | 对 selection 区分度不足 |
| temporal perturbation | 有用但单独不稳定 |
| trajectory slope / late gain | 容易误伤好 checkpoint |

### 5.2 不要把辅助项喧宾夺主

多项指标混合很容易产生“看起来合理但实际错”的分数。

v2.5.3 多个版本说明：

```text
辅助项可以用于异常检测
但不应该随意线性加权决定主排序
```

### 5.3 不要引入过早 checkpoint

ep20 在 v2.5.3i 中明显有害。

早期 checkpoint 可能有较高扰动稳定性或较平滑预测，但这不代表它已经学到适合 DA 的源域结构。

后续 checkpoint bank 至少应从 ep30 开始，甚至可以回到：

```text
30,50,70,100
```

作为更稳的候选池。

### 5.4 tie-break 不能频繁触发

v2.5.3i 中 11/12 触发 tie-break，本质上说明：

```text
tie-break 已经不是 tie-break，而是主选择器
```

如果继续使用 tie-break，触发条件必须极窄，例如：

```text
robust gap <= 0.002 或 0.003
```

否则会破坏 robust 主信号。

### 5.5 fullflow 跑通不等于方法有效

v2.5.3i 完整流程无报错，但 avg 只有 0.6473。

这说明当前瓶颈不是工程流程，而是 selection 信号本身。

## 6. 阶段性结论

v2.5.3 不是一个成功提升版本，但它完成了很重要的假设排除：

```text
静态 zero-shot target-side checkpoint selection
在当前 12 任务设置下信号不足
```

目前可认为：

```text
robust score 是最强的静态 selection 信号
但仍不足以形成稳定、可部署的方法
```

因此，v2.5.3 的合理定位是：

> 负结果版本：证明仅靠无监督静态 target 指标选择 source checkpoint 不够可靠，后续需要引入更接近真实域适应过程的目标域反馈。

## 7. 后续建议

### 7.1 不建议继续做的方向

不要继续投入以下路线：

```text
继续调 confidence / coverage 权重
继续调 blend 权重
继续设计新的静态无监督指标
继续扩大 checkpoint 候选密度
继续加入 ep20 或更早 checkpoint
继续让 tie-break 大范围触发
```

这些方向已经被 v2.5.3 系列基本排除。

### 7.2 可以保留的工程资产

虽然方法失败，但以下代码资产有价值：

```text
checkpoint bank 保存机制
weights_checkpoint 加载接口
selection_metrics_out JSON 输出
selection-only 快速验证模式
robust / perturbation 指标记录
fullflow 脚本框架
```

这些可以作为后续方法的实验基础。

### 7.3 更合理的下一步

后续如果继续 checkpoint selection，应该从静态指标转向 grounded selection。

一个更合理的方向是：

```text
每个 candidate checkpoint 做极短 target warmup
例如 1-2 epoch TimeMatch / pseudo-label adaptation
再根据 warmup 后的 target-side 行为选择
```

可选选择信号：

```text
warmup 后 pseudo-label loss 下降
teacher-student disagreement 下降
target prediction collapse 是否缓解
shift estimate 是否稳定
若允许使用 validation，则直接用 target validation F1
```

这比 zero-shot 指标更接近真实 DA 表现。

### 7.4 最小验证实验

建议只挑 3 个 hard pair 先验证：

```text
32VNH -> 33UVP
33UVP -> 32VNH
31TCJ -> 32VNH
```

候选 checkpoint 不要太密：

```text
30,50,70,100
```

流程：

```text
每个 checkpoint 做 1 epoch target warmup
记录 warmup 后的 target-side 指标
判断是否能选到接近 oracle 的 checkpoint
```

如果这个 grounded selection 仍然不能稳定区分 checkpoint，那么 checkpoint selection 分支应暂时停止。

## 8. 写作叙述建议

对外或写进论文/记录时，不建议把 v2.5.3 表述成“连续失败”。

更合理的叙述是：

```text
我们系统性评估了 target-side zero-shot checkpoint selection 的可行性。
实验表明，静态无监督指标与最终域适应性能之间相关性不足。
这一结果说明 checkpoint selection 需要引入目标域适应过程中的动态反馈，
而不是依赖训练前/短 warmup 后的静态置信度或扰动稳定性。
```

也就是说，v2.5.3 的贡献是：

```text
明确了 checkpoint selection 的困难来源
排除了多类无效启发式
保留了 checkpoint bank 与 selection-only 实验框架
为后续 grounded / adaptive selection 提供依据
```
