# v2.6.1 Phase-Grid Structure View Summary

## Purpose

`v2.6.1` was introduced after the `dir1` and `dir2` GTW-inspired trials. The goal was not to replace the current `v2.4.3b` reshaper backbone, but to test a safer architectural idea:

> keep the `v2.4.3b` residual temporal conv reshaper and original observed-date segment loss, then add an auxiliary phase-grid structure view to reduce dependence on sparse and uneven observed timestamps.

The motivation came from the discussion around GTWIDL and irregular crop time series. Our existing segment partition is tied to observed source-domain dates. That is useful, but it still compares structure on sparse calendar points. The phase-grid view projects each sample onto a fixed latent phase grid before applying the same structure decomposition, giving the reshaper a second, normalized view of temporal structure.

## Architecture Change

Baseline backbone kept from `v2.4.3b`:

- residual temporal conv reshaper
- source-side structure loss
- `doy_gap` segment partition
- trend loss
- segment-inter transition consistency
- boundary-window weighting
- reshaper regularization
- dual-path relation preservation

New in `v2.6.1`:

- `ideas/source_phase_grid.py`
  - projects irregular `[B, T, D]` temporal features to fixed `[B, K, D]` phase slots
  - normalizes positions per sample to `[0, 1]`
  - supports linear / triangular / gaussian kernels
  - uses nearest-fill fallback for low-support phase slots
- optional auxiliary phase-grid structure loss
  - controlled by `--source_phase_grid_trade_off`
  - default disabled unless explicitly set
  - reuses the existing source structure loss on the projected phase-grid features

Important boundary:

> `v2.6.1` does not introduce target structure loss, does not change TimeMatch DA, and does not add a new warp branch to the model. It is an auxiliary source-side structure view.

## Tested Settings

### First quickcheck: K=5, weight=0.10

Setup:

- source pretrain: 50 epochs
- DA: 20 epochs
- checkpoint: epoch 50 only
- phase grid count: 5
- phase-grid trade-off: 0.10

Results:

| Task | F1 |
|---|---:|
| `30TXT -> 33UVP` | 0.7415 |
| `31TCJ -> 32VNH` | 0.5180 |
| `32VNH -> 33UVP` | 0.6470 |
| `33UVP -> 32VNH` | 0.8252 |
| **avg** | **0.6829** |

Interpretation:

- K=5 / weight=0.10 was too coarse and too strong.
- It damaged the two tasks where `v2.4.3b` was already strong.
- The likely reason is that actual `doy_gap` source partitions often produce around 8 intervals, so compressing them to 5 phase slots loses useful boundary detail.

### Second quickcheck: K=8, weight=0.05

Setup:

- source pretrain: 50 epochs
- DA: 20 epochs
- checkpoint: epoch 50 only
- phase grid count: 8
- phase-grid trade-off: 0.05

Results:

| Task | K5 / 0.10 | K8 / 0.05 | Delta |
|---|---:|---:|---:|
| `30TXT -> 33UVP` | 0.7415 | 0.7859 | +0.0444 |
| `31TCJ -> 32VNH` | 0.5180 | 0.5463 | +0.0283 |
| `32VNH -> 33UVP` | 0.6470 | 0.6376 | -0.0094 |
| `33UVP -> 32VNH` | 0.8252 | 0.8238 | -0.0014 |
| **avg** | **0.6829** | **0.6984** | **+0.0155** |

Comparison with `v2.4.3b` on the same four representative tasks:

| Task | `v2.4.3b` | `v2.6.1 K8/w0.05` | Delta |
|---|---:|---:|---:|
| `30TXT -> 33UVP` | 0.8173 | 0.7859 | -0.0314 |
| `31TCJ -> 32VNH` | 0.6862 | 0.5463 | -0.1399 |
| `32VNH -> 33UVP` | 0.6163 | 0.6376 | +0.0213 |
| `33UVP -> 32VNH` | 0.7693 | 0.8238 | +0.0545 |
| **avg** | **0.7223** | **0.6984** | **-0.0239** |

## Conclusion

`v2.6.1` is not currently a replacement for `v2.4.3b`.

The useful signal is more subtle:

- phase-grid helps the same kinds of pairs where warp / phase-aware ideas also helped:
  - `32VNH -> 33UVP`
  - `33UVP -> 32VNH`
- phase-grid still hurts `31TCJ -> 32VNH` badly
- reducing strength and matching the grid count to the actual segment granularity makes the method safer, but not stable enough

This suggests phase-grid is a valid architectural component, but not yet a universal mainline loss.

## Interpretation For The Main Method

The current main method should still be framed around:

> structure-aware temporal reshaper trained by source-side temporal structure decomposition.

`v2.6.1` supports a possible future extension:

> an auxiliary phase-normalized structure view for sparse and uneven time series.

But the experiment also warns that phase normalization can over-regularize when the original calendar-aligned structure is already sufficient. This is consistent with the broader observation from `dir1` and `dir2`: phase / warp ideas are conditionally useful, not globally safe.

## Next Direction

Do not expand `v2.6.1` to 12 tasks yet.

If revisited, the safer route is:

- keep phase-grid as a very weak auxiliary view
- try smaller weights such as `0.02` or `0.03`
- consider enabling it only when phase-grid support is reliable
- avoid using it as a strong replacement for the observed-date segment loss

For the Saturday meeting, `v2.6.1` should be presented as an architectural exploration after `v2.4.3b`, not as the current best result.
