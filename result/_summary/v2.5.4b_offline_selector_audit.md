# v2.5.4b Offline Selector Audit

## 1. 目的

`v2.5.4` 的 metric-bank selector 在 4 个代表任务上只命中 `2/4`，说明继续直接调启发式权重风险很高。

因此 `v2.5.4b` 不再跑 GPU，而是做离线审计：

> 已有 warmup / selection metrics 中，到底有没有足够信息，可以泛化地选出接近 oracle 的 source checkpoint？

这一步的重点不是在已知任务上事后找最优指标，而是检查指标是否具有跨任务泛化能力。

## 2. 实现

新增脚本：

- `analysis/audit_checkpoint_selector_metrics.py`

脚本能力：

- 从 selection 日志中解析 `selection_summary` JSON。
- 对齐 `v2.5.0` checkpoint bank oracle 表。
- 支持 `30 / 50 / 70 / 100` checkpoint 候选。
- 评估单指标 selector：
  - `metric:max`
  - `metric:min`
- 评估指标组 rank aggregation：
  - `robust_perturbation`
  - `distribution_health`
  - `confidence_agreement`
  - `source_prior_shift`
  - `trajectory`
  - `metric_bank_existing`
- 做 leave-one-task-out cross validation。
- 输出 JSON 和 Markdown。

MCP review 提醒了一个关键风险：

> 如果只在同一批任务上选择最优指标，会变成 oracle hindsight，不能证明 selector 能泛化到新任务。

因此脚本加入了 `leave-one-task-out`：

- 留出一个任务。
- 在其余任务上选择表现最好的 metric / group pipeline。
- 用这个 pipeline 去预测留出任务。
- 统计 held-out hit / near-hit / avg gap。

这比普通平均相关性更适合判断是否值得继续做 selector。

## 4. 关键结果

### 4.1 v254-only

同集最优单指标：

| selector | hit | near | avg gap |
|---|---:|---:|---:|
| `selection_max_class_fraction:min` | 2/4 | 2/4 | -0.0345 |
| `metric_bank_rank_score:max` | 2/4 | 2/4 | -0.0438 |
| `selection_temporal_perturbation_score:max` | 2/4 | 2/4 | -0.0438 |

LOTO：

| metric | value |
|---|---:|
| held-out hit | 0/4 |
| held-out near-hit | 0/4 |
| held-out avg gap | -0.1089 |

判断：

> `v2.5.4` 的 metric-bank 在 4 个代表任务上不具备泛化能力。

尤其是：

- `32VNH -> 33UVP` 被选到 `epoch70`，oracle 是 `epoch30`。
- `33UVP -> 32VNH` 被选到 `epoch70`，oracle 是 `epoch30`。

这说明健康分布指标和 robust score 仍会把模型推向“看起来更稳定但迁移更差”的 checkpoint。

## 5. 经验教训

### 5.1 同集表现不能作为 selector 成功依据

`v253i` 中 `selection_score:max` 同集能达到：

```text
hit = 6/12
near = 9/12
avg gap = -0.0069
```

但 LOTO 只有：

```text
hit = 4/12
near = 7/12
avg gap = -0.0262
```

说明同集最优指标存在明显 oracle hindsight。

### 5.2 现有 warmup 指标有弱信号，但不足以独立承担 checkpoint selection

现有指标不是完全无效。

它们能在部分任务上选到 near-oracle checkpoint，例如：

- `30TXT -> 33UVP`
- `31TCJ -> 33UVP`
- `32VNH -> 31TCJ`
- `33UVP -> 31TCJ`

但它们在 hard pair 上不稳定，尤其是：

- `31TCJ -> 32VNH`
- `32VNH -> 33UVP`
- `33UVP -> 32VNH`

这意味着它们可以作为诊断信号，但不能作为主 selector。

### 5.3 指标组聚合没有解决本质问题

`distribution_health`、`confidence_agreement`、`source_prior_shift`、`trajectory` 都没有稳定超过单指标。

这说明失败不是简单的权重问题。

更像是：

> 当前指标没有直接测到“这个 checkpoint 进入完整 TimeMatch 后会不会更好”。

### 5.4 hard pair 是决定方法是否成立的关键

许多任务的 oracle checkpoint 之间差距很小，near-hit 容易看起来不错。

但真正决定 selector 是否有价值的是 hard pair：

- `31TCJ -> 32VNH`
- `32VNH -> 33UVP`
- `33UVP -> 32VNH`

如果这些任务继续错，selector 的平均值再接近也不能说明方法成立。

## 6. 当前结论

`v2.5.4b` 的离线审计结论是：

> **现有 static / short-warmup target metrics 不足以支撑一个可靠的 checkpoint selection 主方法。**

它们有诊断价值，但跨任务泛化不够。

因此不建议继续：

- 调 metric-bank 权重
- 添加更多 confidence / entropy / coverage 变体
- 继续跑 full12 selector-only GPU 实验
- 继续把 checkpoint selection 当作当前唯一主线

## 7. 下一步建议

### 建议路线 A：暂停 selector，转向 source-side structure adaptivity

这是当前更稳的路线。

理由：

- source 有标签，结构评估更可靠。
- checkpoint selector 已连续多轮证明不稳定。
- 源域结构自适应更贴近我们“结构约束自适应性”的逻辑闭环。

下一步可以做：

> `v2.5.5`: source-side structure evaluation gate

第一版不要做复杂神经网络 gate。

更合适的是：

- 计算源域有标签结构评估指标。
- 只调初始结构损失权重。
- 权重调整保守、可解释、有上下界。
- 目标是改善 checkpoint bank 的整体质量，而不是直接替代 target-aware selection。

### 建议路线 B：如果继续 selector，只允许做更强的动态反馈

如果仍要继续 checkpoint selection，就不能再只靠当前静态指标。

下一版必须引入更接近 DA 成败的动态信号，例如：

- warmup 前后 source-target prototype compatibility 的变化
- high-confidence pseudo-label prototype stability 的变化
- teacher-student disagreement 是否在困难类别上下降
- shift estimation 是否收敛到稳定且合理的位置

但这条路线会更慢，也更复杂。

### 当前推荐

> 优先进入 `v2.5.5`，做源域结构自适应；checkpoint selector 暂时降级为辅助分析工具。

这样项目主线更清楚：

- `v2.4.3b`：固定结构 backbone 成立。
- `v2.5.4b`：证明现有目标侧 selector 不足。
- `v2.5.5`：转向更可靠的 source-side adaptivity。
