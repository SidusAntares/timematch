# v2.5.1：checkpoint selection 失败记录与后续教训

## 版本定位

`v2.5.1` 是建立在 `v2.5.0 checkpoint bank + sweep` 之后的一次自动选择尝试。

`v2.5.0` 已经证明：同一个 source-side structure shaping 方法下，不同 source-target pair 偏好的 source checkpoint 不同。因此 `v2.5.1` 试图回答：

> 能否在不使用 target 标签的情况下，通过短程 TimeMatch warmup 和 target-side 无监督指标，自动选择接近 oracle 的 source checkpoint？

它不是结构方法迭代，也不是架构改动，而是 source checkpoint 加载/选择策略。

## 实现方式

`v2.5.1` 的实现逻辑是：

- source-only 阶段复用 `v2.5.0` 产出的 checkpoint bank。
- 候选 checkpoint 为 `epoch30 / epoch50 / epoch70 / epoch100`。
- 每个 checkpoint 先跑短程 TimeMatch warmup。
- warmup 阶段关闭 labeled validation，并跳过最终 labeled test。
- warmup 结束后输出 target-side selection metrics JSON。
- 根据 selection score 自动选择一个 checkpoint。
- 从选中的 checkpoint 启动完整 TimeMatch，并进行最终测试。

主要 selection score 由以下指标组成：

- high-confidence pseudo-label coverage
- mean teacher confidence
- teacher-student agreement
- normalized prediction entropy

这套实现已经跑通，关键问题不在代码链路，而在 selection criterion 本身。

## 选择模式问题

`v2.5.1` 最明显的问题是：它没有真正学到 pair-dependent selection。

实际选择模式为：

| source | selected checkpoints |
|---|---|
| 30TXT | epoch_50 / epoch_50 / epoch_50 |
| 31TCJ | epoch_70 / epoch_100 / epoch_70 |
| 32VNH | epoch_30 / epoch_30 / epoch_30 |
| 33UVP | epoch_100 / epoch_100 / epoch_100 |

这更像 source-dependent selection，而不是 source-target pair-dependent selection。

但 `v2.5.0` 的核心发现恰恰是：最佳 checkpoint 不是 source-global，而是 pair-dependent。

## 为什么 selection score 会误导

从日志中的候选指标看，旧 score 基本奖励：

- coverage 高
- confidence 高
- teacher-student agreement 高
- entropy 低

这些指标能反映“当前 checkpoint 在 target 上预测得是否自信、覆盖是否多”，但不一定反映“后续 DA 是否能走向更好的 target basin”。

几个典型例子：

### 31TCJ->32VNH

- selector 选 `epoch100`
- oracle 是 `epoch50`
- final gap 为 `-0.1944`

warmup 指标：

- selected `epoch100`: score `1.793`, coverage `0.540`
- oracle `epoch50`: score `1.638`, coverage `0.382`

旧 selector 看到 `epoch100` 覆盖更高、score 更高，于是选择它。但完整 DA 结果说明，这种“更自信/覆盖更多”反而可能是错误方向。

### 33UVP->32VNH

- selector 选 `epoch100`
- oracle 是 `epoch30`
- final gap 为 `-0.0699`

warmup 指标：

- selected `epoch100`: score `2.054`, coverage `0.687`
- oracle `epoch30`: score `1.949`, coverage `0.569`

旧 selector 同样倾向于更高 coverage，但 oracle 表明更早 checkpoint 才更适合这个 target。

### 30TXT->33UVP

- selector 选 `epoch50`
- oracle 是 `epoch100`
- final 反而略高于 `v2.5.0` oracle 表中记录

这个任务提醒我们：单次 sweep 和单次 selection 之间也会受到训练随机性影响。因此 selector 不应该只追求命中 checkpoint label，而应该看最终是否接近 oracle performance。

固定 `epoch50`、固定配置时，source self-test 可以较稳定，但 direct target transfer 仍会明显波动。

- selection 算法必须关注 target-specific 信号，而不能只看 source checkpoint 的整体自信程度。

## 失败教训

`v2.5.1` 的失败不是“checkpoint selection 这条路错了”，而是第一版指标设计过粗。

核心教训：

- 不要直接奖励 coverage 越高越好。高 coverage 可能只是错误类别扩张。
- 不要直接奖励 confidence 越高越好。错误高置信会伤害 hard target。
- teacher-student agreement 只能说明两者一致，不说明一致的是对的。
- entropy 低也可能表示 collapse 或 easy-class dominance。
- warmup 5 epoch 可能太短，只看到 early pseudo-label stability，没有看到 adaptation potential。
- 当前 score 更像 source-global 质量分，而不是 source-target pair 适配分。

## 下一版 selector 应该怎么改

下一版不应该简单把旧指标换权重，而应该改变目标：

> 从“选择最自信的 checkpoint”改成“选择最有 target adaptation potential 的 checkpoint”。

建议增加以下约束或指标：

### 1. 类分布健康度

只看 coverage 不够，需要看 pseudo-label 分布是否健康。

可用：

- pseudo-label effective class count
- class distribution entropy
- 与 source prior 的 JS/KL 距离
- high-confidence coverage 的 class-balanced version

目标不是让覆盖率最高，而是避免少数 easy classes 支配 score。

### 2. shift 稳定性

TimeMatch 的 shift 估计本身是 target-aware 信号。

可用：

- AM/IS score 峰值
- top1-top2 shift margin
- shift curve sharpness
- warmup 前后 estimated shift 是否稳定

如果某 checkpoint 的 confidence 高但 shift 不稳定，应该降权。

### 3. 高置信样本结构质量

只在 high-confidence pseudo labels 上计算结构，不直接训练，只作为 selection signal。

可用：

- high-confidence target intra-class compactness
- high-confidence target inter-class separation
- target class prototype balance
- source-target prototype relation consistency

这比全体 target 的 confidence 更接近“伪标签是否可用于后续 DA”。

### 4. warmup 动态，而不是 warmup 终点

旧版主要看 warmup 结束后的静态指标。下一版应该看 5 epoch 内的变化趋势：

- confidence 是否稳步提升
- entropy 是否平滑下降而不是突然 collapse
- agreement 是否提升
- pseudo-label class distribution 是否保持健康
- target shift 是否稳定

目标是选 adaptation trajectory，而不是选 warmup 结束时最漂亮的静态数值。

## 后续验证策略

不建议每次改 selector 都跑 full12。

先用 4 个代表任务做小验证：

- `30TXT->33UVP`
- `31TCJ->32VNH`
- `32VNH->31TCJ`
- `33UVP->32VNH`

判断标准：

- 命中或接近 oracle checkpoint 的比例提高。
- 不再出现同一个 source 对所有 target 都选同一档。
- hard pair 尤其是 `31TCJ->32VNH` 不能再选到明显错误的 `epoch100`。
- 小任务平均性能接近 `v2.5.0` oracle，而不是只接近某个固定 checkpoint。

## 当前结论

`v2.5.1` 应归档为：

> checkpoint selection 路线的第一版失败实现。

它的价值在于：

- 证明了工程链路可行。
- 暴露了 confidence/coverage/agreement/entropy 静态加权 score 的不足。
- 说明下一版必须更 target-aware、更关注伪标签分布健康和 adaptation trajectory。
