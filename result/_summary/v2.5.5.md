# v2.5.5 SVD-Based Source Structure Reliability

## Goal

`v2.5.5` returns to the original `v2.5` goal: source-side adaptivity.

The key change is not a new reshaper branch and not a target-domain gate. Instead, it keeps the `v2.4.3b` backbone and adds a conservative source-side reliability estimator:

> estimate whether each source structural component is stable enough to trust, then mildly adjust the corresponding source structure-loss weights.

## Motivation

Later experiments showed clear task sensitivity:

- fixed `v2.4.3b` structure is useful on average but not universally best
- monotonic / warp / phase-grid variants help some pairs and hurt others
- target-side selector proxies have not been stable enough

Therefore the next step should not be a stronger fixed structure mechanism. It should be a reliability-driven adaptation layer.

GTWIDL provides the conceptual basis: it uses SVD cumulative energy to adapt the number of atoms per class. In our setting, we use the same effective-dimension idea to estimate structure reliability:

- concentrated SVD energy means the structure is stable / compressible
- dispersed SVD energy means the structure is complex or noisy
- stable structure can receive slightly stronger constraints
- dispersed structure should receive weaker constraints

## Implementation

New file:

- `ideas/source_structure_reliability.py`

Main function:

- `compute_svd_structure_reliability_factors`

It computes reliability from source reshaped features:

- intra-segment reliability from class samples within each segment
- trend reliability from class center trajectories
- transition reliability from seasonal residual transitions

Then it maps reliability to conservative factors:

```text
factor = clamp(1 + strength * (reliability - baseline), min_factor, max_factor)
```

Default bounds:

```text
min_factor = 0.70
max_factor = 1.20
strength = 0.35
zeta = 0.90
```

Affected weights:

- `source_structure_intra_trade_off`
- `source_structure_trend_trade_off`
- `source_structure_segment_inter_trade_off`
- `source_structure_boundary_window_trade_off`

The adaptation is disabled by default and only enabled when:

```bash
--source_structure_adaptive_weights True
--source_structure_adaptivity_mode svd_reliability
```

## Boundary

This version:

- does not use target labels
- does not add target structure loss
- does not add a neural gate
- does not change TimeMatch DA logic
- does not change the reshaper architecture
- does not enable phase-grid by default

It only changes source structure-loss weights during source pretraining.

## Quickcheck Launcher

New launcher:

```bash

```

Default quickcheck:

- four representative tasks
- source pretrain 50 epochs
- DA 20 epochs
- one source model per task
- SVD reliability adaptivity enabled

Run:

```bash
cd /data/user/timematch
nohup bash launchers/launch_four_gpu_v255_svd_reliability_quickcheck.sh > logs/v255_svd_reliability_quickcheck_nohup.out 2>&1 &
```

## Expected Reading

Success does not require every task to improve immediately. The first question is whether SVD reliability reduces the damage from over-trusting unstable structure components while preserving the gains of `v2.4.3b`.

The most important pairs to watch first:

- `31TCJ -> 32VNH`
- `32VNH -> 33UVP`
- `33UVP -> 32VNH`

If this works, the next step is to tune only the conservative mapping range, not to introduce a high-capacity gate.
