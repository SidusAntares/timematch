# v2.5.0

## 1. 版本定位

`v2.5.0` 是 `v2.4.3b` 结构方法之上的**验证版本**。

如果说：

- `v2.4.3b` 已经证明：
  - `DOY-gap-aware segment`
  - `residual + trend + weak inter-segment`
  - `boundary window` 作为 **weighting / saliency signal**
  是当前有效的结构方法

那么 `v2.5.0` 要回答的问题就是：

> **同一个 source-side structure shaping 方法，最佳 source checkpoint 是否依赖具体 source-target 对。**

因此，`v2.5.0` 的核心不是改结构，而是做：

> **checkpoint bank + sweep**

也就是：

- source-only 训练到 `100 epoch`
- 保存 `epoch 30 / 50 / 70 / 100` 四个 checkpoint
- 对每个 source-target 任务分别从这四个 checkpoint 启动 TimeMatch
- 观察不同任务偏好的 source checkpoint 是否不同

---

## 2. 实现说明

### 2.1 结构方法保持不变

`v2.5.0` 继续沿用 `v2.4.3b`：

- `source_segment_partition_mode = doy_gap`
- `source_structure_loss_version = segment_boundary_window_residual`
- boundary window 只做 `inter-segment` 的 weighting / saliency signal

不改：

- 主干架构
- temporal unit 定义
- boundary window 的基本角色

---

### 2.2 新增 checkpoint bank

对每个 source-only 训练过程，保存多个 checkpoint：

- `epoch_30.pt`
- `epoch_50.pt`
- `epoch_70.pt`
- `epoch_100.pt`

然后对每个 target 任务，分别从这些 checkpoint 进入完整 `TimeMatch`。

因此，`v2.5.0` 实际上是在估计：

> **pair-dependent best source initialization**

---

### 2.3 配置口径

`v2.5.0` 使用：

- 结构主干：`v2.4.3b`
- source-only pretrain：`100 epoch`
- saved checkpoints：`30 / 50 / 70 / 100`
- `TimeMatch = 20 epoch`
- `num_workers = 8`

---

## 4. oracle 最优结果

如果对每个任务都能选到最优 checkpoint，那么 12 个任务的 oracle 平均 `macro_f1` 为：

- **`0.6891`**

这比固定单一 checkpoint 的平均结果明显更高：

- `epoch30 avg = 0.6633`
- `epoch50 avg = 0.6562`
- `epoch70 avg = 0.6422`
- `epoch100 avg = 0.6544`

也就是说：

> **pair-dependent checkpoint selection 本身具有显著潜力。**

---

## 5. 结果分析

### 5.1 最优 checkpoint 明显不是统一的

四个候选 checkpoint 的任务最优分布为：

- `epoch30` 最优：4 个任务
- `epoch50` 最优：3 个任务
- `epoch70` 最优：4 个任务
- `epoch100` 最优：1 个任务

这说明：

> **固定一个全局 source-only epoch 已经不合理。**

不同 source-target 对确实偏好不同训练深度的 source checkpoint。

---

### 5.2 source-side structure shaping 确实存在“训过头”的问题

从多个任务可以看到：

- 有些任务明显偏好较早 checkpoint
- 继续把 source 训练到更后期，反而伤害 transfer

例如：

- `31TCJ -> 32VNH`
  - `epoch50 = 0.6957`
  - `epoch100 = 0.5526`

- `33UVP -> 32VNH`
  - `epoch30 = 0.7765`
  - `epoch100 = 0.7047`

这说明：

> **当前问题不再是 source-side structure shaping 有没有用，而是做到哪一步最适合当前 target。**

---

### 5.3 但也不是“越早越好”

`v2.5.0` 同时说明：

- 并不是所有任务都偏好更早 checkpoint
- 仍有任务偏好更晚 checkpoint

例如：

- `30TXT -> 33UVP`
  - `epoch100 = 0.7637` 最优

因此当前不能简单下结论说：

> “以后统一用 `50 epoch` 或统一用 `30 epoch` 就行。”

更合理的结论是：

> **最佳 source checkpoint 是 pair-dependent 的。**

---

## 6. 当前结论

`v2.5.0` 可以概括为：

### 已经验证成立的

- checkpoint bank + sweep 是有价值的
- 最佳 source checkpoint 明显依赖 source-target 对
- 固定单一 source-only epoch 已经不再合理

### 当前主线意义

`v2.5.0` 不是最终方法版，而是：

> **为 `v2.5.1` 提供直接证据的验证版。**

它已经说明：

> **下一步确实值得做 target-aware source checkpoint selection。**

也就是说，`v2.5.1` 的任务不再是验证“需不需要选 checkpoint”，而是：

> **如何在不看 target 标签的前提下，自动选出接近最优的 source checkpoint。**
